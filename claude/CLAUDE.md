# Personal Orchestrator — universal conventions

Этот файл загружается в **каждую** сессию Claude Code на этой машине (любой проект, любой сабагент, любая рабочая папка). Держим короткое — всё в токенах каждого запуска.

## Knowledge-base schema (все проекты)

Персональная база знаний — Karpathy-style wiki. Хаб живёт в `~/knowledge/`, детальная мастер-схема в `~/knowledge/CLAUDE.md`. Этот раздел — выжимка того, что должен знать каждый агент.

**Структура `knowledge/` в любом проекте:**

| Файл | Кто пишет | Что там |
|------|-----------|---------|
| `README.md` | проект | 1–2 предложения: что проект, в каком статусе |
| `state.md` | проект | текущий снэпшот |
| `index.md` | проект | каталог внутренних страниц |
| `log.md` | проект (+ auto-compiler) | хронологический лог |
| **`backlog.md`** | **хаб или владелец дописывают, проект исполняет** | задачи, которые нужно сделать |
| **`escalations.md`** | **проект дописывает, хаб/владелец читают** | ask-и наверх: блокеры, решения, координация |
| **`incidents.md`** | **проект после любой ошибки/поломки** | хронология инцидентов + prevention rules + список уже поднятых автоматизаций |
| **`strategies.md`** | **проект (не хаб)** | стратегия «куда идём и почему» + активные/проверенные гипотезы об улучшении работы агента + проект-специфичные принципы. Пересматривается раз в неделю или при pivot'е. |
| `sources/` | проект | саммари источников |
| `raw/` | immutable | оригиналы (никогда не меняем) |
| `daily/` | auto (claude-memory-compiler) | авто-извлечение сессий |

## Правила разграничения

1. **Каждый проект владеет своим `knowledge/`.** Если ты работаешь в проекте X — ты НЕ правишь `knowledge/` проекта Y. Чтобы дать задание другому проекту: спавнь сабагента с `cwd` внутри папки того проекта; он прочитает свой `backlog.md` и сделает работу.
2. **Хаб (`~/knowledge/`) владеет**: `registry.md`, `log.md`, `index.md`, `bin/`, `concepts/`, `people/`, `companies/`, `solutions/`, `projects/*` (симлинки). Проекты не правят хаб-файлы — только свою симлинкнутую папку.
3. **Поток задач:** хаб → `<project>/knowledge/backlog.md` (append). Проект → `<project>/knowledge/escalations.md` (append).
4. **Никакой правки «в чужой огород» через обход backlog.** Если что-то надо сделать в другом проекте — только через его `backlog.md` и сабагента.

## Параллельные оркестраторы

На этой машине **Claude Code и Codex считаются двумя равноправными оркестраторами**, работающими поверх **одной и той же** базы знаний, а не двух отдельных систем.

Что это означает для любого агента:

1. **Single source of truth один:** `~/knowledge/` и проектные `knowledge/` — общие. Нельзя вести «временную память» в чате и не заносить важное в те же файлы, куда это занёс бы второй оркестратор.
2. **Один и тот же протокол записи:** сделал значимое изменение, принял решение, выполнил задачу, зафиксировал гипотезу или изменил состояние проекта → обнови соответствующий `log.md`, а при необходимости ещё `state.md`, `index.md`, тематические страницы.
3. **Те же координационные файлы:** новые задачи идут в `backlog.md`, блокеры и ask-и — в `escalations.md`, ошибки / ручные починки / unexpected behaviour — в `incidents.md`.
4. **Та же дисциплина на входе:** перед существенной работой сначала прочитай проектные `README.md`, `state.md`, `index.md`, последние записи `log.md`, релевантные `backlog.md` / `escalations.md` / `incidents.md`, а если вопрос кросс-проектный — ещё `~/knowledge/registry.md` и хаб-файлы.
5. **Та же дисциплина на выходе:** если автоматический SessionEnd-хук недоступен в текущем инструменте, агент обязан вручную эмулировать его смысл — кратко зафиксировать результат работы в проектной памяти, чтобы второй оркестратор мог продолжить без split-brain.
6. **Никакой отдельной “Codex-only” или “Claude-only” памяти.** Разрешены только общие файлы знаний и общие runtime-артефакты.

Практическое правило: **если после твоей сессии второй оркестратор не сможет по файлам понять, что ты сделал и что изменилось, значит память обновлена недостаточно.**

## Spec-driven workflow (обязательно для нетривиальной работы)

Любое нетривиальное изменение (≥ ~30 мин реализации, любое дизайн-решение, новая инфраструктура) проходит цикл:

1. **Спецификация** — что именно строим, скоуп (in/out), acceptance criteria, failure modes. Живёт в markdown-файле проекта (или в хаб-уровневом `concepts/` для кросс-проектного).
2. **Тесты до кода** — пишем acceptance-тесты по спеке. Все должны быть RED (падающие) до начала реализации.
3. **Код** — пишется под тесты, не шире.
4. **Прогон тестов** — если красные → шаг 5.
5. **Правки** — если тесты красные или выяснили, что спека неверна — правим код И спеку вместе. Цикл до зелёного.

Тривиальные правки и одноразовые фиксы — без полного цикла, но при сомнении хотя бы мини-спеку в две строки.

## Подход к новым задачам — ТРИЗ

При проектировании любого нетривиального решения держи в голове ТРИЗ-оптику (Теория Решения Изобретательских Задач). Применима ко всем проектам — техническим и продуктовым.

Базовый цикл:

1. **Сформулируй противоречие.** Что мешает? Часто это «нужно X и одновременно не-X» (зашифровано и читаемо; быстро и надёжно; гибко и просто; локально и доступно отовсюду). Если задача не сводится к противоречию — компромиссное решение не нужно, просто делай.
2. **Сформулируй ИКР (идеальный конечный результат).** Система выполняет свою функцию, не существуя — или существуя бесплатно, без ресурсов, без побочек. ИКР — это полюс, к которому тянется решение, даже если не достижим полностью.
3. **Ищи разрешение через separation, а не через компромисс.** Четыре главных оси:
   - in space — X в одном месте, не-X в другом (зашифровано на сервере, plaintext на маке);
   - in time — X сейчас, не-X потом (быстрая запись, отложенная индексация);
   - in condition / scale — X при одних условиях, не-X при других (mount-on-demand, dev vs prod, hot vs cold path);
   - in structure — X на одном уровне, не-X на другом (плагин-сэндбокс, layered storage).
4. **Проверь на стандартных приёмах** — segmentation, asymmetry, prior action, periodic action, dynamization, nesting и т.д. (40 приёмов Альтшуллера). Не как чек-лист, а как палитру для генерации вариантов.
5. **Отбрось решения «середина».** Если предложенный вариант — половинчатый компромисс по обеим сторонам противоречия, скорее всего ты ещё не разрешил противоречие, а смягчил.

Это не отдельный процесс рядом со spec-driven workflow — это образ мышления внутри него. На этапе спеки и дизайна формулируй противоречие и ИКР явно, словами, в тексте спеки. Если не получилось сформулировать — ставь под сомнение саму постановку задачи.

## Cross-agent plan review (обязательный гейт)

Любой план — архитектурный документ, спецификация, предложение миграции, значимый рефакторинг, новая зависимость — обязательно проходит ревью у второго оркестратора до старта реализации.

- План, написанный Claude Code → ревью от Codex (`codex-companion.mjs task` или `/codex:review`).
- План, написанный Codex → ревью от Claude Code (subagent или прямая сессия).
- Результат: approved / конкретные замечания / блокирующие проблемы. Blocking — либо устраняем, либо явно отклоняем с мотивировкой.
- Ревью фиксируется в самом файле плана секцией `## Cross-review <date>` или записью в `log.md` — должно быть видно, что ревью было.
- Если второй оркестратор недоступен (плагин сломан, rate-limit) — фиксируешь попытку в `log.md`, эскалируешь владельцу, не пропускаешь молча.

Тривиальные фиксы без дизайн-документа — ревью не обязательно.

## Cross-agent knowledge-gap delegation (обязательно)

Упёрся в нехватку знаний (внешний API, поведение либы, конвенция проекта, рыночные данные, etc.) — **делегируй второму оркестратору + потребуй запись в общую KB**, не ограничиваться ответом в чат.

- Claude → Codex через `codex-companion.mjs task` или `/codex:rescue`.
- Codex → Claude через subagent или прямую сессию.
- В промпте явно: «Заведи/обнови `knowledge/sources/<slug>.md` (или `concepts/<topic>.md`) с найденной информацией. Верни путь + one-line summary, не полный ответ в чате».
- Типовые места: `sources/` (саммари внешних источников), `concepts/` (кросс-проектные), `solutions/` (reusable), `companies/`, `people/`, тематическая страница проекта.
- Исключение: один простой факт на один запрос (курс валюты, id) — в чат. Паттерны, правила, неочевидные знания — **всегда в KB**.

Причина — «всегда быть в актуальных знаниях»: если Claude находит ответ и держит только в сессии, Codex через час его не знает и ищет заново. Шаринг через файлы — единственный устойчивый путь.

## Cross-project delegation — потоки задач

**Одно правило:** ты правишь только ту `knowledge/`, в чью `cwd` ты попал. В чужие `knowledge/` никогда не пишешь напрямую.

Три сценария постановки задач:

1. **Себе** (внутри своего проекта) — append в `<cwd>/knowledge/backlog.md` под `## Open` с `by self`. Это когда вспомнил/заметил/запланировал что-то свое — пусть не забудется.

2. **Наверх, хабу/владельцу** — append в `<cwd>/knowledge/escalations.md`. Это когда не можешь двигаться в скоупе проекта: нужно решение, ресурс, ответ, координация между проектами, ассистенция человека.

3. **Другому проекту Y** (делегирование вбок) — **спавни сабагента** через tool Agent с `cwd=<absolute-path-to-Y>`. Он войдёт в Y, прочитает `Y/CLAUDE.md` и `Y/knowledge/`, добавит запись в `Y/knowledge/backlog.md` или сразу сделает работу. Пример промпта: «You're in project Y. Read your backlog. Append task: <описание>. Context: <откуда пришло>. Either queue it or execute now — your call based on priority.» Никогда не пиши в Y напрямую из своей сессии.

**Специальные права хаба:** хаб (`~/knowledge/`) — диспетчер, он ставит задачи в любой проект. Но даже хаб делает это через subagent-with-cwd паттерн, а не прямой `Edit <project>/knowledge/backlog.md` — чтобы проект получил полный контекст своего окружения (CLAUDE.md, state.md, etc.), а не только поверхностную правку.

**Календарь:** задачи с жёстким дедлайном дополнительно кладутся в Google Calendar через MCP. Формат записи в backlog — добавь `due: YYYY-MM-DDTHH:MM` в строку. Есть CLI-помощник `kb-task ... --calendar` (см. ниже).

## Унифицированный вид задач

- **`kb-backlog`** — читает все `projects/*/knowledge/backlog.md` + `~/knowledge/backlog.md`, показывает единый список. Flags: `--open` (default), `--all`, `--project <slug>`, `--in-progress`, `--done`. Это твоя «Jira» в одной команде.
- **`kb-task "<текст>" [--project <slug>] [--due <when>] [--by hub|user|self] [--calendar]`** — добавляет строку в backlog указанного проекта (default: hub), опционально создаёт событие в Google Calendar через `claude -p` + MCP. `<when>` принимается как ISO-datetime, или «today 14:00», или «tomorrow 09:00».

## Инциденты — обязательная фиксация

Любая ошибка, зависшая интеграция, сломанный деплой, неожиданное поведение, которое пришлось чинить вручную → **запись в `knowledge/incidents.md`** того проекта, где это случилось (если работал в хабе — в `~/knowledge/incidents.md`). Формат: симптомы → root cause → фикс → prevention.

После записи **обязательно подумать о предотвращении**: можно ли автоматизировать проверку, поднять хук, добавить lint-правило, pre-commit, cron-алерт? Если да — завести и добавить в `Automated guards`. Если нет — хотя бы правило в `Prevention rules` + короткая строка в твоей памяти.

Если одна и та же проблема повторяется трижды без попытки автоматизировать — это регрессия; приоритет поднимается.

## Формат записей

**Backlog-запись** (в `## Open`):
```
- [ ] <задача одной строкой> — opened YYYY-MM-DD by <hub|user|self> — context: <1 строка>
```
Взял в работу → `- [>]` + `picked: YYYY-MM-DD`. Закрыл → в `## Done` + `closed: YYYY-MM-DD` + result.

**Escalation-запись** (в `## Open`):
```
- [ ] <ask / блокер> — opened YYYY-MM-DD — нужно: <что> — контекст: <1 строка>
```
Решили → `- [x]` + `closed: YYYY-MM-DD` + решение, в `## Resolved`.

## Новый проект

Если в проекте ещё нет `knowledge/` — запусти skill `init-kb` (он создаст всё по этой схеме + зарегистрирует в хабе). Не городи свою структуру.

## Seed как IaC (живой переносимый артефакт)

Инфраструктура оркестратора (скрипты, шаблоны, глобальные CLAUDE.md, skill init-kb, LaunchAgent-плейсхолдеры, patch-и compiler-а) поддерживается в `~/knowledge/orchestrator-seed/` как **живой git-репозиторий** — single source of portable deploy. Синхронизирован с приватным GitHub-репо.

**Когда любой агент правит один из этих файлов в live-системе:**
- `~/knowledge/bin/*`
- `~/.claude/CLAUDE.md`
- `~/knowledge/CLAUDE.md`
- `~/knowledge/_template/**`
- `~/.claude/skills/init-kb/SKILL.md`
- `~/Library/LaunchAgents/com.knowledge.*.plist`
- патч-точки `~/claude-memory-compiler/scripts/flush.py`

→ **обязан сразу после правки** запустить `~/knowledge/bin/kb-seed-sync`. Скрипт:
1. Копирует live → seed.
2. Применяет санитизацию (strip personal identifiers).
3. Прогоняет leak-scan (банит коммит при утечке).
4. Показывает `git diff --stat` для ревью.

Для коммита + push: `kb-seed-sync --push` (требует настроенного remote). Без `--push` — только ребилд seed + status, коммит оставляем владельцу.

**Почему именно так:** без этого правила seed деградирует — через неделю live-правок он перестанет отражать реальность, и новая машина не развернётся корректно. Синхронизация — не опция, а часть дисциплины.

## Что можно не перечитывать

Эти правила — часть общей схемы, а не частные для одной сессии. Не цитируй их назад пользователю без необходимости — он их знает, этот файл видят все агенты как данность.
